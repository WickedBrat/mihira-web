# Native Sign In with Apple ‚Äî Design

**Date:** 2026-07-11
**Scope:** Replace the web-based (Safari) Apple OAuth flow with native Sign In with Apple on iOS, still authenticating through the existing Supabase Auth setup. Android/web keep the current web OAuth flow.

---

## 1. Goals

- iOS users tapping "Continue with Apple" get the native Face ID/Touch ID Apple sign-in sheet instead of a Safari popup.
- The resulting session is a normal Supabase session (via `signInWithIdToken`), indistinguishable downstream from the existing `signInWithOAuth` sessions ‚Äî `useAuth`/`useUser`/`useSession` in `lib/auth.tsx` need no changes.
- Android, web, and any iOS device where native Apple auth is unavailable keep working exactly as today (`signInWithOAuth('apple')` via `WebBrowser`).
- No changes required to `OAuthButton`, `ProfileAuthSheet`, the onboarding screens, or `oauth-native-callback.tsx` ‚Äî they all call `signInWithApple()` from `useSignIn`, which is the only place branching logic lives.

## 2. Non-Goals

- Not switching to Apple's official `AppleAuthenticationButton` component ‚Äî the existing custom `OAuthButton` already satisfies Apple HIG (logo + "Continue with Apple" + adequate contrast).
- Not touching the Google OAuth path.
- Not configuring the Supabase Apple provider or the Apple Developer "Sign In with Apple" App ID capability ‚Äî both are assumed already done (per user confirmation). If either isn't actually configured, the native exchange will fail with a Supabase auth error surfaced through the existing toast error path ‚Äî no silent failure.

## 3. New Dependencies

- `expo-apple-authentication` ‚Äî native Sign In with Apple module (autolinked via existing `use_expo_modules!` in `ios/Podfile`).
- `expo-crypto` ‚Äî SHA-256 hashing + random bytes for the OIDC nonce.

## 5. Error Handling

- **User cancels the native sheet** (`error.code === 'ERR_REQUEST_CANCELED'`): return `false` silently, no error toast (matches how a user dismissing Apple's own UI shouldn't be treated as an app error).
- **Any other native error, or Supabase exchange error**: caught by the same `catch` block pattern already in `useSignIn`, logged via `console.error`, surfaced via the existing `showToast({ type: 'error', ... })` ‚Äî same UX as today's OAuth failures.
- **`identityToken` missing**: treated as a generic failure (throws, caught, toast shown) ‚Äî should not happen in practice per Apple's API contract but guards against a malformed credential.
- **`updateUser` full-name patch fails**: logged only, does not block sign-in or show a toast ‚Äî it's a cosmetic enhancement, not critical path.

## 7. Testing / Verification

- `pod install` after adding `expo-apple-authentication`, confirm it links without error.
- Manual verification on a physical iOS device or simulator signed into an Apple ID (native Sign In with Apple does not work in Expo Go ‚Äî requires the dev client, which this project already uses given the bare/native setup):
  - Tap "Continue with Apple" ‚Üí native sheet appears (not Safari).
  - Successful auth ‚Üí session created, profile screen shows signed-in state.
  - Cancel the native sheet ‚Üí no error toast, button returns to idle state.
- Confirm Android/web path is untouched by reading the diff ‚Äî `signInWithProvider` body stays as-is, only reached via the fallback branch.

## 8. Out of Scope

- Apple's official `AppleAuthenticationButton` UI component.
- Changes to the Google sign-in path.
- Supabase Apple provider dashboard configuration (assumed already done).
- Apple Developer "Sign In with Apple" App ID capability configuration (assumed already done ‚Äî if EAS-managed credentials are in use, EAS will offer to sync this automatically on the next build once the entitlement is present).
